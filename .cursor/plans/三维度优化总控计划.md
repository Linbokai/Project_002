# 买量脚本工作台 — 三维度优化总控计划

> 本文档基于 `三维度项目优化建议_b42fcef2.plan.md` 分析报告，是所有优化任务的执行指挥中心。
> 每个模块完成后更新状态和备注。

---

## 项目概况

- **技术栈**: Vue 3 + TypeScript + Pinia + TailwindCSS 4 + Vite 7
- **代码规模**: 137 个文件（82 .ts + 55 .vue），~12,000 行
- **架构**: MVVM（View → ViewModel → Model）
- **关键大文件**: config-panel.vue(806), script-templates.ts(804), gameplay-prompt-builder.ts(345), use-chat.ts(342), api-config-dialog.vue(338), script-shot-renderer.vue(336), use-video-analysis.ts(313)

## 模块总览与依赖关系

```
Phase 1（基础清理）── 无外部依赖，优先执行
  ├─ R1 消除重复代码（P0 程序层）     ← 触及 composables
  └─ R3 品牌与主题化（P0 美术层）     ← 触及 UI/样式
      两模块无共享文件冲突，可并行

Phase 2（架构 + 核心功能）── 依赖 Phase 1
  ├─ R2 架构解耦（P1 程序层）         ← 依赖 R1（use-chat.ts）
  ├─ R5 脚本版本管理（P0 功能层）     ← 独立
  └─ R6 数据闭环（P0 功能层）         ← 独立
      R5、R6 可并行；R2 在 R1 之后串行

Phase 3（体验完善）── 各自独立
  ├─ R4 动效与排版（P1 美术层）
  ├─ R7 批量生成与导入导出（P1 功能层）
  ├─ R8 合规预检与校验（P1 功能层）
  └─ R9 评分可视化增强（P1 美术层）
      均可并行

Phase 4（长期优化）── P2
  ├─ R10 泛型化与拆分（P2 程序层）
  └─ R11 移动端与无障碍（P2 美术层）
      可并行
```

### R2: 架构解耦 `P1` `程序层`

**状态**: 🟢 已完成
**优先级**: P1
**预估改动**: 6 个现有文件 + 1 个新文件

**问题描述**:
1. settings-store 内部依赖 model-routing-store（Store 跨层依赖）
2. variant-store 直接引用 service 层的 `getAngleLabel`
3. use-chat 依赖 10 个 Store，过于臃肿
4. `getModelForTask` auto 模式下 resolved 为 undefined 时无 fallback
5. use-model-routing 静默吞错，model-discovery-api 错误信息不足

**涉及文件**:
| 文件 | 操作 |
|------|------|
| `src/composables/use-resolved-model.ts` | 🆕 新建（桥接 settings + model-routing） |
| `src/stores/settings-store.ts` (89行) | ✏️ 移除 model-routing-store 依赖 |
| `src/stores/variant-store.ts` (81行) | ✏️ 移除 service 依赖 |
| `src/utils/variant-labels.ts` 或 `src/constants/` | ✏️ 移入 getAngleLabel |
| `src/composables/use-chat.ts` (342行) | ✏️ 抽取 buildGenerationContext 纯函数 |
| `src/composables/use-model-routing.ts` (76行) | ✏️ 增加错误处理 |
| `src/services/api/model-discovery-api.ts` (20行) | ✏️ 增加错误解析 |

**具体任务**:
1. 新建 `use-resolved-model.ts`：封装 `getModelForTask` 逻辑，内部桥接 settings-store + model-routing-store
2. settings-store：移除 `getModelForTask` 中对 `useModelRoutingStore()` 的调用，改为纯 switch（仅读自身 config）
3. 所有调用 `settingsStore.getModelForTask()` 的地方改为调用 `useResolvedModel()` composable
4. variant-store：将 `getAngleLabel` 移到 `src/constants/variant-angles.ts`
5. use-chat：抽取 `buildGenerationContext()` 纯函数到 `src/services/helpers/`，减少 Store 直接依赖
6. use-model-routing：catch 中增加 `console.error` + toast 提示
7. model-discovery-api：使用 `parseApiError` 解析响应体
8. `getModelForTask`（无论在哪）增加 default 分支 fallback
9. `pnpm build` 验证

**与其他模块的依赖**: 依赖 R1 完成（R1 改动 use-chat.ts）
